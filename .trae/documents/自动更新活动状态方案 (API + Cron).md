用户经过考虑，认为“动态生成任务”的方案过于复杂，决定回归**“通过 API 定时调用更新状态”**的方案。

这是一个明智的选择，因为批量扫描更新（Polling/Cron）在 Web 系统中远比单体动态任务更健壮、更易维护，且不易出错。

**最终实施计划：**

1.  **创建定时任务 API**
    *   **文件**: `src/app/api/cron/update-event-status/route.ts`
    *   **逻辑**:
        *   查询所有 **未完成** 的活动（状态为 `PAID` 或 `ACTIVE`）。
        *   **批量检查**：
            *   若 `当前时间 >= 开始时间` 且状态是 `PAID` -> 变更为 `ACTIVE`。
            *   若 `当前时间 >= 结束时间` 且状态是 `ACTIVE` (或 `PAID`) -> 变更为 `COMPLETED`。
        *   **批量更新**：将变更写回数据库。
        *   **响应**：返回更新了多少条记录。

2.  **提供调用指南**
    *   虽然不能帮用户配置外部 Cron（如 Linux crontab），但我会在代码注释或 README 中说明如何调用此 API。
    *   **为了方便用户验证**，我会在“我的活动”页面或 Dashboard 加一个（仅开发可见或手动触发的）隐藏按钮或逻辑，或者告知用户直接访问该 URL 即可触发更新。

3.  **可选优化：在关键页面触发**
    *   虽然用户说“通过 API 定时调用”，但为了体验更好，我可以在 `src/app/[locale]/(organizer)/my-events/page.tsx` (活动列表页) 的 `useEffect` 中，**顺便**异步调用一次这个 API（非阻塞）。
    *   这样用户在访问管理页时，数据总是最新的，给人一种“自动更新”的感觉，同时也支持外部定时器调用。
