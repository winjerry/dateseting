I have found the issue!

**The Root Cause:**
The `getServerSession` function in `src/core/auth/session.ts` is currently returning a **MOCK SESSION** with a hardcoded user ID: `'mock-organizer-id'`.

```typescript
export async function getServerSession() {
  // Mock session for development
  // ...
  return {
    user: {
      id: 'mock-organizer-id', 
      // ...
    }
  };
}
```

However, the real authentication system (BetterAuth) is creating *actual* users in the database with real UUIDs (e.g., generated by `getUuid()`).
When `src/app/api/events/checkout/route.ts` runs:
1.  It calls `getServerSession()` and gets the mock user with ID `'mock-organizer-id'`.
2.  It then calls `findUserById('mock-organizer-id')`.
3.  The database (obviously) does not contain a user with ID `'mock-organizer-id'` (unless one was manually inserted).
4.  `findUserById` returns `undefined`.
5.  My defensive check triggers: "User record not found..."

**Why did this happen?**
It seems the project is in a transition state or development mode where the session handling was mocked, but now we are trying to use real DB features (foreign keys) that require real data integrity.

**The Fix:**
I need to update `src/core/auth/session.ts` to actually use BetterAuth to retrieve the *real* session from the request headers, instead of returning a mock object.

**Plan:**
1.  Modify `src/core/auth/session.ts` to use `getAuth()` and retrieve the real session using `auth.api.getSession`.
2.  I will need to import `headers` from `next/headers` to pass the request headers to BetterAuth so it can verify the session cookie.

This will ensure `getServerSession` returns the currently logged-in user (who actually exists in the DB), satisfying the foreign key constraint.
